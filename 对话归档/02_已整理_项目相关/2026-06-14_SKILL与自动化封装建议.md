# SKILL 与自动化封装建议

## 主题

基于近期项目推进方式，识别适合长期复用的工作流封装，并落成项目专用 Skill、子代理定义和收工检查脚本。

## 背景

近期工作中，反复出现以下模式：

- 按 `00_总控主线 + 01~05 支线` 推进项目
- 每轮工作前看 `README / 更新日志 / 待办清单`
- 把零散讨论回收到主文档或对话归档
- 工作结束时判断是否要更新日志、待办并同步到 GitHub

这些动作输入相对稳定、重复成本高，而且当前只有部分被同步脚本覆盖。

## 关键结论

- 高优先级封装对象不是单点写作，而是“项目总控推进习惯”本身。
- 最适合先落地的形式是项目专用 Skill，用来固化你的工作节奏、输出结构和判断规则。
- 对单个专题的深度收敛，适合再补一个专题收敛子代理定义。
- 双机同步已经有脚本，但“收工前检查日志、待办、归档、冲突风险”的动作还没有被覆盖，适合补一个收工检查脚本。

## 可执行动作

- 使用 [skills/julong-project-copilot/SKILL.md](../../skills/julong-project-copilot/SKILL.md) 作为后续项目主 Skill
- 使用 [topic-review-subagent.md](../../skills/julong-project-copilot/references/topic-review-subagent.md) 作为专题委派模板
- 在 Mac 或 Windows 结束一轮工作后，先运行根目录收工检查脚本，再决定是否执行一键同步

## 是否需要回收进主文档

- 不需要

本次结论已经直接落到了 `skills/`、`README.md` 和 `更新日志.md`。
