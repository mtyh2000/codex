# 平台与对接方案

## 适用范围

本文件用于印尼聚龙集团棕榈果一期项目的平台与对接分支评估，重点回答平台应记录什么、如何形成园区到工厂的数据闭环、一期需要哪些统计报表、以及工时通或既有平台需要确认哪些接口。

当前默认前提：

- 园区网络覆盖率约 60%，不能依赖全程在线。
- 一期先跑通一类、二类果实的分级与计数闭环。
- 手持终端需要支持本地识别、本地缓存、有网补传。
- 平台需要形成园区、地块、车辆、工厂之间的统计与溯源。
- 本文件只聚焦一期平台闭环，不展开长期智慧农业模块。

## 一期平台目标

一期平台不追求一次性建设完整智慧农业大平台，建议先围绕三类能力落地：

1. 记录：完整接收并保存终端识别数据、原始图片、地块归属、设备人员、车辆与工厂验收信息。
2. 统计：按园区、地块、车辆、工厂、时间维度统计一类、二类果实数量、占比、上传状态和验收差异。
3. 对接：与工时通或既有平台打通人员、车辆、定位、调度、组织地块等必要数据，支撑运输联动和数据归档。

平台一期的核心价值是把“园区装果 -> 车辆运输 -> 工厂收果”的数据链路先跑通，保证每条识别结果能回溯到图片、地块、人员、设备和验收结果。

## 平台核心业务对象

### 主数据对象

| 对象 | 一期用途 | 关键说明 |
| --- | --- | --- |
| 园区 | 数据归属、权限范围、统计维度 | 需确认是否已有园区编码 |
| 地块 | 产量统计、调度触发、溯源定位 | 需确认地块边界、编号、所属园区 |
| 工厂收果点 | 到厂验收、二次识别、比对统计 | 可先按工厂或收果口管理 |
| 操作人员 | 拍照、装车、验收责任追溯 | 需确认与工时通人员档案是否同步 |
| 手持终端 | 采集设备、算法版本、上传来源 | 需绑定终端编号、设备状态、模型版本 |
| 车辆 / 卡车 | 运输任务、调度联动、到厂比对 | 需确认车辆编号和工时通定位能力 |
| 车斗 / 装载单元 | 判断装载批次和调度触发 | 若现场按车斗管理，需单独建档 |
| 果实等级 | 一类、二类数量与占比 | 一期先按 2 类，保留扩展字段 |

### 业务单据对象

| 单据 | 触发场景 | 平台作用 |
| --- | --- | --- |
| 采摘 / 收果识别记录 | 园区现场拍照识别后生成 | 保存图片、数量、等级、地块归属 |
| 图片记录 | 每次识别或复核产生 | 支撑追溯、复核、训练、争议处理 |
| 装车 / 批次记录 | 多条识别记录归并到一次装车 | 关联地块、车辆、司机、预计到厂 |
| 调度任务 | 达到装载或运输条件后生成 | 推送司机或同步既有调度系统 |
| 工厂验收记录 | 车辆到厂后人工或 AI 二次识别 | 与园区数据做差异比对 |
| 异常 / 复核记录 | 数据差异、图片异常、上传失败 | 形成闭环处理和责任追溯 |

## 平台应记录字段

### 1. 园区现场识别记录

每次手持终端拍照识别后，平台应生成一条识别记录。建议字段如下：

| 字段 | 说明 | 一期建议 |
| --- | --- | --- |
| 识别记录 ID | 平台唯一编号 | 必填，由平台生成 |
| 终端本地记录 ID | 终端离线时生成的唯一编号 | 必填，用于补传去重 |
| 终端编号 | 采集设备编号 | 必填 |
| 操作人员 ID / 姓名 | 拍照文员或现场操作人 | 必填，先支持手选或登录带出 |
| 拍照时间 | 终端本地拍照时间 | 必填 |
| 上传时间 | 数据到达平台时间 | 必填，补传时与拍照时间不同 |
| 园区编码 / 名称 | 所属园区 | 必填 |
| 地块编码 / 名称 | 所属地块 | 必填 |
| GPS / 北斗坐标 | 拍照位置 | 建议必填，弱信号时允许为空并标记 |
| 图片原图地址 | 原始图片存储地址 | 必填 |
| 图片缩略图地址 | 列表展示用 | 建议必填 |
| 果子总数量 | AI 输出总数 | 必填 |
| 一类果数量 | AI 输出分类数量 | 必填 |
| 二类果数量 | AI 输出分类数量 | 必填 |
| 识别置信度 | 整体或分类置信度 | 建议保留 |
| 模型版本 | 本次识别使用的模型版本 | 必填 |
| App 版本 | 终端应用版本 | 建议保留 |
| 是否离线生成 | 是否无网状态下产生 | 必填 |
| 上传状态 | 未上传、上传中、已上传、失败、已补传 | 必填 |
| 关联车辆 / 批次 | 后续装车后绑定 | 可后补 |
| 备注 | 现场补充说明 | 可选 |

关键要求：终端本地记录 ID、终端编号、拍照时间三者应能支持补传去重，避免弱网环境下重复上传造成统计偏差。

### 2. 图片留存字段

图片不应只作为列表缩略图使用，应作为追溯、复核、模型训练和争议处理的数据资产保留。

| 字段 | 说明 | 一期建议 |
| --- | --- | --- |
| 图片 ID | 图片唯一编号 | 必填 |
| 关联识别记录 ID | 对应哪次识别 | 必填 |
| 图片类型 | 原图、压缩图、工厂图、复核图 | 必填 |
| 存储地址 | 文件或对象存储地址 | 必填 |
| 图片拍摄时间 | 终端记录时间 | 必填 |
| 上传时间 | 平台接收时间 | 必填 |
| 文件大小 | 便于上传与存储管理 | 建议保留 |
| 图片哈希 | 去重、防篡改、追溯 | 建议保留 |
| 拍摄位置 | GPS / 北斗坐标 | 建议保留 |
| 水印信息 | 时间、地点、终端编号等 | 建议保留 |
| 是否进入训练集 | 是否用于后续模型训练 | 建议保留 |
| 复核状态 | 未复核、已确认、需重拍、有争议 | 建议保留 |

一期建议至少保留原图和缩略图。原图用于复核和训练，缩略图用于后台列表和移动端快速查看。

### 3. 装车 / 运输批次字段

若现场按车斗或车辆组织运输，建议平台把多条园区识别记录归并为一个运输批次。

| 字段 | 说明 | 一期建议 |
| --- | --- | --- |
| 批次 ID | 装车或运输批次唯一编号 | 必填 |
| 园区 / 地块 | 批次来源 | 必填，可支持多地块但一期建议先单地块 |
| 关联识别记录列表 | 本批次包含哪些拍照记录 | 必填 |
| 批次总数量 | 汇总果子总数 | 必填 |
| 一类果汇总数量 | 汇总一类果 | 必填 |
| 二类果汇总数量 | 汇总二类果 | 必填 |
| 车斗编号 | 若有车斗管理则记录 | 待确认 |
| 车辆编号 | 承运车辆 | 必填 |
| 司机 | 承运司机 | 必填 |
| 装车时间 | 现场装车时间 | 必填 |
| 发车时间 | 离开园区时间 | 建议保留 |
| 预计到厂时间 | 调度预估 | 建议保留 |
| 目的工厂 / 收果点 | 到厂验收地点 | 必填 |
| 调度任务 ID | 关联调度任务 | 建议保留 |
| 批次状态 | 待装车、运输中、已到厂、已验收、异常 | 必填 |

一期建议先按“单地块到单车辆到单工厂收果点”的简单口径设计。如果实际存在多地块混装，应在评估期单独确认混装规则，否则会影响溯源准确性。

### 4. 工厂验收比对字段

工厂端可以先支持人工录入验收结果，也可以接入摄像头或终端二次识别。平台字段应兼容两种方式。

| 字段 | 说明 | 一期建议 |
| --- | --- | --- |
| 验收记录 ID | 工厂验收唯一编号 | 必填 |
| 关联批次 ID | 对应哪次运输批次 | 必填 |
| 车辆编号 | 到厂车辆 | 必填 |
| 工厂 / 收果点 | 验收地点 | 必填 |
| 到厂时间 | 车辆到厂时间 | 必填 |
| 验收时间 | 完成验收时间 | 必填 |
| 验收方式 | 人工录入、手持识别、固定摄像头识别 | 必填 |
| 工厂图片地址 | 到厂图片或视频截图 | 建议必填 |
| 工厂果子总数量 | 工厂端确认数量 | 必填 |
| 工厂一类果数量 | 工厂端分类数量 | 必填 |
| 工厂二类果数量 | 工厂端分类数量 | 必填 |
| 园区上报总数量 | 从批次带出 | 必填 |
| 园区一类果数量 | 从批次带出 | 必填 |
| 园区二类果数量 | 从批次带出 | 必填 |
| 总数差异 | 工厂数 - 园区数 | 必填 |
| 一类差异 | 工厂一类 - 园区一类 | 必填 |
| 二类差异 | 工厂二类 - 园区二类 | 必填 |
| 差异率 | 差异数量 / 园区数量 | 建议保留 |
| 比对结论 | 正常、超阈值、需复核 | 必填 |
| 复核人 / 复核时间 | 异常处理记录 | 异常时必填 |

一期比对不宜只看总数，还应同时看一类、二类数量差异，因为分级差异会直接影响后续质量评价和工厂出油判断。

### 5. 终端与模型版本字段

平台需要知道每条数据来自哪个设备和哪个算法版本，否则后续准确率评估和问题追溯会缺少依据。

| 字段 | 说明 | 一期建议 |
| --- | --- | --- |
| 终端编号 | 设备唯一编号 | 必填 |
| 终端状态 | 在线、离线、异常、停用 | 必填 |
| 绑定人员 | 当前使用人 | 建议保留 |
| 所属园区 | 默认使用范围 | 建议保留 |
| App 版本 | 应用版本 | 必填 |
| 模型版本 | AI 模型版本 | 必填 |
| 最近识别时间 | 最近一次拍照 | 建议保留 |
| 最近上传时间 | 最近一次成功上传 | 建议保留 |
| 待补传数量 | 本地未上传记录数 | 建议保留 |
| OTA 状态 | 可升级、升级中、失败、已完成 | 建议保留 |

## 数据闭环流程

### 1. 园区采集

拍照文员在园区现场选择园区、地块，对堆放或待装车棕榈果进行拍照。终端本地运行 AI 模型，输出果子总数、一类果数量、二类果数量，并生成本地识别记录。

平台设计要求：

- 园区和地块必须在采集时确定，不能只在后台事后人工猜测。
- 终端应支持按人员权限或默认园区减少误选。
- 图片、识别结果、时间、终端编号应一起保存。

### 2. 弱网缓存与补传

无网或弱网时，终端继续本地识别并缓存记录；有网后自动补传数据和图片。

平台设计要求：

- 接口必须支持断点补传和失败重试。
- 平台必须根据本地记录 ID 做幂等处理，重复补传不能重复计数。
- 上传状态要可见，后台能查看哪些记录已上传、哪些还在终端待补传。
- 统计报表应区分“已上传数据”和“可能仍有待补传数据”，避免弱网造成误判。

### 3. 装车与调度触发

平台按地块汇总识别数量，当某地块或车斗累计数量达到标定车辆容量或现场调度条件时，生成运输调度建议。

平台设计要求：

- 调度触发条件一期建议先支持人工确认，避免模型数量或现场混装导致误派车。
- 若接入工时通定位，可根据车辆位置、车辆状态、司机状态筛选最近可用车辆。
- 调度任务应绑定地块、批次、车辆、司机、目的工厂。

### 4. 车辆运输

车辆接收调度任务后运输到工厂。平台记录任务状态、发车时间、预计到厂时间、到厂状态。

平台设计要求：

- 一期如工时通已有车辆定位能力，平台只需读取车辆位置和状态，不必重复建设车载定位系统。
- 若车辆定位接口暂不可用，先支持人工选择车辆和司机，并保留对接字段。

### 5. 工厂验收与比对

车辆到厂后，工厂端对果串进行二次识别或人工验收录入，平台将工厂验收结果与园区上报批次做比对。

平台设计要求：

- 工厂验收必须关联到运输批次，否则无法形成闭环。
- 比对维度至少包括总数、一类数量、二类数量。
- 差异超过阈值时生成异常记录，支持查看园区图片、工厂图片、人员、车辆和时间。

### 6. 统计、溯源与复核

平台最终形成地块产量、等级占比、车辆运输、工厂验收差异、异常复核等数据。

平台设计要求：

- 任一统计数字应能下钻到原始识别记录和图片。
- 任一工厂差异应能追溯到对应地块、批次、车辆、司机、园区拍照记录和工厂验收记录。
- 异常复核结果应回写，避免只发现问题但没有闭环处理。

## 一期统计报表

### 1. 园区 / 地块统计

建议指标：

- 今日、本周、本月果子总数量。
- 一类果数量、二类果数量。
- 一类果占比、二类果占比。
- 地块排名和趋势。
- 已上传记录数、待补传记录数、上传失败记录数。
- 异常识别或需复核记录数。

用途：

- 判断各地块收果进度。
- 辅助安排采摘人员、拍照文员和运输车辆。
- 形成地块级产量与质量基础台账。

### 2. 终端采集统计

建议指标：

- 每台终端拍照次数。
- 每名操作人员采集记录数。
- 平均识别数量。
- 离线生成记录数。
- 补传成功率。
- 图片异常或重拍次数。
- App 版本、模型版本分布。

用途：

- 验证手持终端在真实环境下是否稳定。
- 识别人员操作差异和培训问题。
- 追踪模型版本对识别结果的影响。

### 3. 运输调度统计

建议指标：

- 待调度批次数。
- 已生成调度任务数。
- 车辆承运批次数。
- 从装车到到厂耗时。
- 车辆空闲、运输中、异常状态。
- 按司机 / 车辆统计运输次数和承运数量。

用途：

- 支撑车辆就近调度。
- 评估运输效率。
- 为后续接入油耗、工时、维修等数据保留基础。

### 4. 工厂验收比对统计

建议指标：

- 到厂批次数。
- 已验收批次数。
- 园区上报总数与工厂验收总数。
- 一类、二类数量差异。
- 差异率。
- 超阈值异常批次数。
- 异常复核处理状态。

用途：

- 形成园区到工厂的闭环监管。
- 发现运输、装车、识别或验收环节的差异。
- 支撑争议处理和质量复盘。

### 5. 图片与数据质量统计

建议指标：

- 原图留存数量。
- 图片上传成功率。
- 图片缺失记录数。
- 低置信度识别记录数。
- 需人工复核图片数。
- 可进入训练集样本数。

用途：

- 支撑模型持续训练。
- 保障追溯材料完整。
- 提前发现拍照角度、光照、遮挡等现场操作问题。

## 待确认接口清单

### 1. 工时通 / 既有平台主数据接口

需要确认：

- 是否已有园区、地块、工厂、人员、车辆、司机的统一编码。
- 平台主数据由谁维护，是工时通维护还是新平台维护。
- 是否允许新平台读取主数据。
- 主数据同步方式是接口拉取、定时同步、人工导入，还是双方都可维护。
- 人员、车辆、地块停用或变更后如何同步。

一期建议：

- 若既有平台已有主数据，优先沿用既有编码，避免新旧系统口径不一致。
- 若暂时没有接口，先支持 Excel 或后台维护导入，但字段要预留外部编码。

### 2. 终端上传接口

需要确认：

- 终端数据上传协议和鉴权方式。
- 图片与识别结果是一次上传，还是先上传记录再上传图片。
- 单张图片大小限制、压缩规则、失败重试机制。
- 是否需要支持批量补传。
- 终端本地记录 ID 的生成规则。
- 上传失败后终端保留多久，是否允许人工重新触发上传。

一期建议：

- 接口必须支持幂等写入。
- 图片上传与识别记录应能分开重试。
- 平台要返回明确的上传成功、重复、失败原因。

### 3. 车辆定位与状态接口

需要确认：

- 工时通是否能提供车辆实时位置。
- 是否能提供车辆在线状态、空闲状态、运输中状态。
- 是否能提供司机与车辆绑定关系。
- 车辆定位刷新频率是多少。
- 弱网情况下车辆位置延迟如何标记。

一期建议：

- 若定位接口可用，平台用于“推荐车辆”，调度最终由人员确认。
- 若定位接口不可用，一期先人工选择车辆和司机，保留接口字段。

### 4. 调度任务推送接口

需要确认：

- 调度任务推送到哪里：工时通、司机 App、短信、后台列表，还是人工电话通知。
- 是否需要司机确认接单。
- 是否需要回传发车、到厂、完成状态。
- 调度任务失败或司机拒绝时如何处理。
- 是否需要导航路线下发。

一期建议：

- 先把调度做成“平台生成建议 + 人工确认派单”。
- 对接成熟后，再升级为自动推荐最近车辆并推送司机。

### 5. 工厂验收接口

需要确认：

- 工厂端一期是人工录入、手持终端拍照，还是固定摄像头识别。
- 工厂识别结果是否使用同一套模型。
- 工厂是否按车辆、批次、车斗或收果口验收。
- 工厂端图片和视频截图是否需要留存。
- 验收差异阈值由谁定义。
- 异常复核由园区处理、工厂处理，还是双方共同处理。

一期建议：

- 字段先兼容人工验收和 AI 二次识别。
- 必须要求验收记录绑定运输批次。
- 比对异常要形成复核状态，而不是只显示差异。

### 6. 数据大屏 / 管理系统推送接口

需要确认：

- 是否需要给集团数据大屏推送一期统计结果。
- 推送哪些指标：地块产量、等级占比、车辆运输、工厂验收差异等。
- 推送频率是实时、定时，还是日报级。
- 数据大屏是否需要下钻到原始图片。

一期建议：

- 先输出标准统计接口，保证数据口径稳定。
- 大屏展示先以汇总指标为主，图片下钻可在管理平台内完成。

## 异常与复核机制

一期平台至少需要覆盖以下异常：

| 异常类型 | 触发条件 | 处理建议 |
| --- | --- | --- |
| 上传失败 | 终端数据或图片多次上传失败 | 终端继续缓存，平台展示待补传数量 |
| 重复上传 | 同一终端本地记录 ID 重复提交 | 平台幂等处理，不重复统计 |
| 图片缺失 | 有识别结果但原图未成功上传 | 标记为图片待补传，不进入最终追溯闭环 |
| 地块缺失 | 识别记录未选择地块 | 不允许提交或进入待补录状态 |
| 低置信度 | 模型置信度低于阈值 | 标记需人工复核 |
| 批次未绑定车辆 | 已装车但未关联车辆 | 不允许进入运输中状态 |
| 工厂未验收 | 车辆到厂后长期未验收 | 提醒工厂或调度管理员 |
| 验收差异超阈值 | 园区与工厂数量或等级差异过大 | 生成异常复核单 |
| 混装不清 | 一个批次来源多个地块但未拆分 | 标记溯源风险，需人工确认 |

## 权限与角色

一期建议先设置以下角色：

| 角色 | 主要权限 |
| --- | --- |
| 拍照文员 | 终端拍照识别、选择园区地块、查看本人上传状态 |
| 园区管理员 | 查看本园区地块统计、识别记录、异常、补传状态 |
| 调度员 | 查看待运输批次、生成或确认调度任务、绑定车辆司机 |
| 司机 | 接收调度任务、反馈发车和到厂状态 |
| 工厂验收员 | 创建验收记录、录入或确认二次识别结果 |
| 集团管理员 | 查看跨园区、跨工厂统计、验收差异和异常闭环 |
| 系统管理员 | 维护主数据、终端、模型版本、接口配置 |

如果一期用户体系复杂度受限，可先把“园区管理员、调度员、工厂验收员、集团管理员”作为后台角色，把拍照文员和司机放在移动端或既有平台中处理。

## 一期页面建议

后台平台一期建议至少包含：

1. 识别记录列表：按园区、地块、时间、人员、终端、上传状态筛选，支持查看原图和识别结果。
2. 地块统计看板：展示地块日、周、月收果数量，一类、二类数量与占比。
3. 待调度批次：展示达到装载或运输条件的地块 / 批次，支持生成调度任务。
4. 调度任务列表：展示车辆、司机、批次、目的工厂、状态和时间节点。
5. 工厂验收比对：展示园区上报数、工厂验收数、差异和复核状态。
6. 图片复核中心：查看低置信度、图片缺失、异常差异相关图片。
7. 终端管理：查看设备状态、模型版本、App 版本、最近上传、待补传数量。
8. 主数据管理：维护园区、地块、人员、车辆、工厂收果点等基础资料。

## 数据口径建议

### 地块归属口径

- 每条园区识别记录必须归属到一个明确地块。
- 一期建议优先避免多地块混装；如现场无法避免，需要在批次上记录多个来源地块及各自数量。
- 若地块来自既有平台，应保留既有地块编码和平台内部 ID。

### 数量统计口径

- 园区统计以手持终端识别记录为基础。
- 工厂统计以验收记录为基础。
- 闭环统计以“已绑定运输批次且已完成工厂验收”的记录为基础。
- 弱网情况下，应区分已上传记录和终端待补传记录。

### 图片留存口径

- 原图应尽可能完整保留，不仅保留缩略图。
- 识别记录没有原图时，不应视为完整可追溯记录。
- 工厂验收若产生图片或截图，应与园区图片共同用于差异复核。

### 比对口径

- 比对对象应是同一运输批次，而不是仅按日期或车辆模糊匹配。
- 比对维度至少包括总数、一类数量、二类数量。
- 差异阈值需要客户确认，评估期可先作为配置项保留。

## 当前已确认与待确认

### 已确认信息

- 园区网络覆盖约 60%，平台和终端必须按弱网设计。
- 一期先做一类、二类果实识别和业务闭环。
- 手持终端需支持图片留存、计数、分类、本地缓存、有网补传。
- 平台需接收终端数据，并按园区、地块或工厂展示和统计。
- 方案方向包含园区装果、车辆运输、工厂收果的闭环。
- 车辆调度可考虑依托工时通定位能力。

### 待确认问题

1. 园区、地块、人员、车辆、工厂是否已有统一编码体系。
2. 拍照口径是按堆、按车斗、按车辆，还是按采摘批次。
3. 现场是否允许多地块混装；如果允许，如何拆分数量归属。
4. 工厂验收一期采用人工录入、手持终端识别，还是固定摄像头识别。
5. 工厂验收差异阈值如何定义。
6. 工时通能否开放车辆定位、车辆状态、司机绑定、调度任务推送接口。
7. 调度任务是平台内确认，还是必须推送到司机端。
8. 图片原图需要保留多久，是否有存储容量或合规要求。
9. 弱网补传失败时，现场人员和后台管理员分别如何处理。
10. 平台是否需要与集团数据大屏同步一期汇总指标。

## 一期实施建议

建议平台分三步推进：

1. 先建采集入库能力：终端上传接口、图片留存、识别记录、园区地块归属、上传状态。
2. 再建闭环能力：装车批次、车辆绑定、调度任务、工厂验收、差异比对。
3. 最后建统计与对接能力：地块统计、终端统计、验收差异统计、工时通和数据大屏接口。

评估阶段最应优先确认的是主数据编码、现场拍照批次口径、工厂验收方式和工时通接口边界。这四项一旦明确，平台字段、页面、接口和数据闭环就能比较稳定地进入原型和研发设计。
