AI复核与样本闭环部署建议

一、问题背景

在“PAD应用程序与端侧算法闭环”里，最下面那几块能力通常包括：
- 后台复核
- 视觉大模型复核
- 标准样本库
- 模型训练与发布

这里需要明确一个关键问题：

这些能力应该放在什么地方？

是放在SAP里，
还是单独放一台高性能主机，
还是租用云服务器，
还是采用混合部署？


二、核心结论

不建议把这些能力放在SAP里。

第一阶段也不建议为了这些能力专门采购一台高性能主机做全本地部署。

最优方案通常是：

PAD端：
负责本地离线识别

独立业务后台 / AI中台：
负责后台复核、样本库、模型管理、模型发布、大模型调用

SAP：
只接收最终确认后的结算结果

如果现场网络特别差，可以加一个：
工厂本地轻量缓存节点

也就是说，推荐结构是：

PAD端离线识别
+ 云端业务后台 / AI中台
+ 工厂本地缓存节点（可选）
+ SAP只接结果


三、为什么不建议放在SAP里

1. SAP适合做结果系统，不适合做AI平台

SAP更适合承接：
- 主数据
- 财务结果
- 成本中心
- 结算结果
- 凭证或业务结果归档

SAP不适合承接：
- 图片存储
- 大量样本管理
- 模型训练
- 模型版本发布
- 视觉大模型异步调用
- 误判样本回流
- PAD端模型下发


2. 改造复杂度高

如果把AI复核和样本闭环能力直接塞进SAP，会导致：
- 接口复杂
- 实现成本高
- 变更周期长
- 后续迭代困难


3. 灵活性不够

样本规则、模型版本、复核流程和大模型供应商选择都可能不断变化。

这些变化适合在独立业务后台中快速迭代，不适合在SAP里频繁变更。


4. 容易影响主业务稳定性

SAP的核心任务是结算和业务结果归档。

如果把图片、模型、样本、复核都压进去，容易把正式业务系统拖复杂，影响稳定性。


四、为什么第一阶段不建议专门买高性能主机

1. 第一阶段目标是验证闭环，不是建设本地AI算力中心

第一阶段更重要的是：
- PAD端能不能离线识别
- 后台能不能对账和复核
- 结算能不能跑通
- 样本能不能回流

而不是先搭一套重型本地算力平台。


2. 大模型通常更适合通过API调用

第一阶段视觉大模型一般用于：
- 低置信度样本复核
- 刺猬果疑似样本复核
- 争议样本复核
- 抽样质检样本复核

这类调用不是每时每刻都大量发生，更适合：
- 后台任务队列
- 异步API调用
- 云端模型服务


3. 本地高性能主机增加维护负担

如果自购高性能主机，会额外增加：
- 硬件采购成本
- 散热要求
- 供电要求
- 备份要求
- 故障维护
- 系统升级
- 本地IT依赖

对第一阶段试点来说，通常不划算。


4. 训练也不需要一直本地跑

模型训练不是全天连续高频工作。

更合理的方式通常是：
- 由算法公司训练
- 或使用云GPU训练
- 训练完成后只下发PAD端轻量模型


五、推荐部署方案

1. 推荐方案：云端业务后台 + PAD离线识别

最推荐的方式是：

PAD端：
- 端侧识别算法
- 离线缓存
- 采收批次管理

云端业务后台：
- 采收对账主数据库
- 图片对象存储
- 后台复核
- 样本库
- 模型版本管理
- 模型下发
- 视觉大模型复核任务

SAP：
- 接收确认后的结算结果

优点：
- 架构清晰
- 维护相对轻
- 扩展方便
- 更适合接大模型API


2. 可选方案：本地缓存节点 + 云端主后台

如果客户担心印尼网络波动，可以在工厂或园区加一个本地轻量节点。

这个节点的职责是：
- 缓存PAD补传数据
- 缓存图片
- 在局域网内先收数据
- 再择机同步云端主后台

这个节点不建议承担：
- 主样本库
- 主模型训练
- 主AI平台
- SAP主接口

它更适合作为：
缓存、转发、补传中转节点


3. 不推荐方案：全部做在本地高性能主机

除非客户明确要求本地化、且具备长期运维能力，否则不建议第一阶段全部压到本地主机。

不推荐原因：
- 成本高
- 运维难
- 故障影响大
- 模型服务扩展不灵活
- 初期风险高


六、什么情况下才考虑本地高性能主机

只有以下情况，才建议认真考虑本地高性能主机：

1. 客户明确要求数据完全本地化。
2. 网络差到连云端后台都经常不可用。
3. 视觉大模型必须私有化部署。
4. 图片和训练量足够大，长期算下来本地算力更划算。
5. 客户本地IT团队有能力长期维护。

如果这些条件不具备，就不建议第一阶段投入高性能本地主机。


七、如果使用 ZGX Nano，应该怎么定位

ZGX Nano 可以用，但定位要摆正。

它更适合做：
- 本地AI增强节点
- 疑难样本复核节点
- 样本库相关AI处理节点
- 模型实验或小规模本地推理节点

它不适合直接替代：
- PAD端现场采集终端
- 采收对账主数据库
- 地磅对账后台
- SAP结果系统

更准确地说，ZGX Nano 适合放在：

PAD端：
负责现场采集、离线识别和弱网缓存

业务后台：
负责采收对账、复核、结算和SAP输出

ZGX Nano：
负责本地AI增强能力

也就是：

PAD负责现场
+ 业务后台负责主账
+ ZGX Nano负责本地AI增强
+ SAP只接最终结果


1. 为什么 ZGX Nano 可以考虑

如果客户希望：
- 尽量本地化
- 本地也具备较强AI能力
- 对云端大模型依赖不要太高
- 疑难样本能在本地做更强复核

那么 ZGX Nano 这类设备是可以考虑的。


2. 为什么不建议把它当唯一核心节点

ZGX Nano 的强项是AI算力，而不是业务系统承载。

这个项目除了AI，还有很多核心业务模块：
- 采收批次主账
- 地磅导入
- 自动对账
- 异常复核
- 结算确认
- SAP输出
- 权限和审计

这些模块更适合放在标准业务后台和数据库上，而不是全部压在ZGX Nano上。


3. 为什么不建议用 ZGX Nano 直接替代 SAP

SAP适合做结果系统。

ZGX Nano 适合做AI增强。

两者职责不同：
- SAP负责最终业务结果归档和财务接口
- ZGX Nano 负责本地视觉AI能力

把 ZGX Nano 当成 SAP 替代物，会让系统边界混乱。


4. 如果采用 ZGX Nano，推荐怎么放

推荐方式一：
云端业务后台 + PAD离线识别 + ZGX Nano本地AI增强节点

适合：
- 既想保留云端主后台
- 又想让部分AI能力本地化


推荐方式二：
本地业务服务器 + ZGX Nano + SAP外部结果系统

适合：
- 客户明确要求本地化
- 本地IT有能力维护
- 网络环境长期不稳定


不推荐方式：
把 ZGX Nano 作为唯一主机，既承载业务后台，又承载AI，又承载SAP中间层。

原因：
- 职责混乱
- 故障影响面大
- 运维风险高


5. 推荐结论

如果客户问“ZGX Nano 能不能用”，更稳妥的回答是：

可以用，但应把它当作本地AI增强节点，而不是整个系统的唯一核心主机。

一句话总结：

ZGX Nano 适合放在 AI 增强层，
不适合替代 PAD，
不适合替代业务后台，
也不适合替代 SAP。


八、各能力推荐放置位置

1. PAD端

放置内容：
- 登录和采收应用
- 离线缓存
- 本地识别算法
- 本地模型版本记录
- 同步队列


2. 独立业务后台 / AI中台

放置内容：
- 采收对账主数据库
- 图片对象存储
- 后台复核
- 样本库
- 模型版本管理
- 模型发布
- 大模型异步复核任务


3. SAP

放置内容：
- 最终确认后的结算结果
- 财务或业务归档结果

不建议放置：
- 图片
- 样本库
- 模型文件
- 大模型调用
- 模型训练


九、最终推荐架构

推荐最终架构如下：

PAD端：
本地离线识别

↓

后台主库：
采收批次、图片、复核、样本库、模型管理

↓

地磅对账与结算：
异常复核后生成结算结果

↓

SAP：
只接收最终结果

同时：

后台主库
↓
视觉大模型复核
↓
专家确认
↓
样本回流
↓
算法公司训练
↓
新模型下发PAD


十、最终建议

这几块能力：
- 后台复核
- 大模型复核
- 样本库
- 模型发布

不建议放在SAP里。

第一阶段也不建议为了它们单独采购一台高性能主机做全本地化。

最优方案是：

独立业务后台 / AI中台
+ PAD端离线识别
+ SAP只接最终结果
+ 必要时增加本地缓存节点

一句话总结：

SAP负责结果，
PAD负责现场，
独立后台负责AI闭环，
而不是把所有东西都塞进SAP或本地主机里。
